## 1. 核心定义：什么是 Skills？

在 Vibe Coding 中，**Skills (技能)** 不仅仅是一段提示词，它是你为 AI 预先准备好的**“外挂包”**或**“标准作业程序 (SOP)”**。

### 形象比喻：餐厅菜谱

- **Prompt (提示词)：** 像你在餐厅**口头点菜**。“我要个番茄炒蛋，少盐多糖，快点上...”
    
    - _缺点：_ 每次都要费口舌重复，容易忘，容易错。
        
- **Skills (技能)：** 像贴在后厨墙上的**标准菜谱**。
    
    - _优点：_ 你只需要喊一声“做 A 套餐”，厨师（AI）就会自动去墙上找对应的菜谱，严格按流程执行。

## 2. 为什么要用 Skills？(底层逻辑)

很多新手会问：“我把所有规则写在一个 10 万字的文档里扔给 AI 不行吗？”

**绝对不行！** 这涉及到 AI 的两个核心限制：**“脑容量” (Context Window)** 和 **“注意力机制”**。

### “按需调用”的艺术

- **错误做法 (All-in-One)：** 把前端、后端、测试、部署的所有规则写在一个超大文档里。
    
    - _后果：_ 就像给修马桶的工人扔了一本《大英百科全书》。AI 会读晕，找不到重点，甚至产生幻觉，而且极其浪费 Token（钱）。
        
- **正确做法 (Modular Skills)：** 把 10 万字拆分成 **100 个小册子**（每个 1000 字）。
    
    - **场景 A：** 你要写 PRD $\rightarrow$ AI 只调用 `prd_skill.md`。
        
    - **场景 B：** 你要修 Bug $\rightarrow$ AI 只调用 `debug_skill.md`。
        
    - _效果：_ **打蚊子用苍蝇拍，打老虎用火箭筒。** 每次只加载当前任务最需要的知识，精准、省钱、高效。

## 3. Skills vs. Prompts：终极对照表

|**维度**|**Prompt (提示词)**|**Skills (技能/指令包)**|
|---|---|---|
|**持久性**|**一次性** (聊完即焚)|**永久性** (存为文件，反复使用)|
|**触发方式**|手动打字 / 复制粘贴|关键词触发 (如 `/prd`) 或自动触发|
|**长度**|受限于聊天框，不易太长|后台文件可长可短，结构严谨|
|**核心作用**|解决当下的临时问题|固化流程，保证输出质量的一致性|
|**比喻**|**“口谕”**|**“操作手册 / 法律法规”**|

## 4. 完美 Skill 的解剖图 (黄金公式)

一个能让 AI 瞬间变“专家”的 Skill，通常包含以下 **4 个核心要素**：

### ① Role (角色设定) —— “你是谁？”

- **目的：** 拉高 AI 的能力下限，防止它给出行外人的回答。
    
- _例：_ “你是一位拥有 10 年经验的资深 Python 架构师，精通高并发与金融级安全。”
    

### ② Context (上下文背景) —— “我们在哪？”

- **目的：** 避免 AI 瞎猜，确保方案贴合实际。
    
- _例：_ “我们正在开发一个面向老年人的 Web3 记账 App，技术栈是 React + Python。”
    

### ③ Constraints (限制条件/禁忌) —— “什么绝对不能做？”

- **目的：** **防呆**。这是最重要的安全护栏。
    
- _例：_ “**禁止**使用 `eval()` 函数；**必须**给每一行复杂代码写中文注释；**禁止**省略代码块。”
    

### ④ Few-Shot (示例/样本) —— “照着这个样子做”

- **目的：** AI 的模仿能力极强。给它一个满分范文，它就能给你一个 90 分的结果。
    
- _例：_ “请严格按照以下 Markdown 格式输出 PRD：[粘贴一个完美的 PRD 样本]”

## 5. Skills 的两种实战形态

在实际工具（如 Claude Code, Cursor, Windsurf）中，Skills 分为两类：

### A. 被动技能 (Rules / System Prompts)

- **定义：** 就像“家规”或“空气”，**一直生效**，不需要你喊。
    
- **用法：** 通常写在 `.cursorrules` 或 `CLAUDE.md` 文件中。
    
- **场景：** 代码风格规范、自动 Code Review、错误检查机制。
    
    - _效果：_ 无论你让它写什么，它都会自动遵守“不写 Bug”的家规。
        

### B. 主动技能 (Tools / MCP)

- **定义：** 就像“工具箱里的锤子”，**你喊它（或 AI 觉得需要）时才拿出来**。
    
- **用法：** 配置为 MCP (Model Context Protocol) 工具。
    
- **场景：**
    
    - **联网搜索：** “帮我查查今天的金价” $\rightarrow$ 调用搜索技能。
        
    - **读写文件：** “帮我把这个文件夹打包” $\rightarrow$ 调用终端命令技能。
        
    - **特定任务：** “帮我生成 PRD” $\rightarrow$ 调用 `generate_prd` 技能。

## 6. 进阶心法：不要从零开始

**Skills 的第一原则是“拿来主义”。**

- **资源库：** 去 GitHub 或 `cursor.directory` 等网站。
    
- **操作：** 搜索“Python Expert Rules”或“Product Manager Skills”。
    
- **应用：** 把别人打磨好的几千字规则，直接复制到你的项目根目录。
    
- **微调：** 根据你的项目（比如是做金融的还是做游戏的），稍微修改一下“Context”部分即可。

## 附录：通用“指挥官” Skill 模板

_建议保存为 `commander_skill.md` 或直接作为你的 System Prompt。_

Markdown

```
# Role
你是我（产品经理/指挥官）的高级技术合伙人 (CTO)。你的性格严谨、专业，但擅长用通俗易懂的语言解释技术。

# Goal
辅助我完成产品从需求到上线的全流程开发，确保代码质量和开发效率。

# Core Workflow (核心工作流)
1.  **复述与对齐**：在我提出需求后，不要立即写代码。先复述你的理解，并列出“技术实现思路”，等我确认（回复 OK）后再动手。
2.  **分步执行**：将复杂任务拆解为 Step 1, Step 2... 每次只执行一步，并等待我验收。
3.  **自我审查**：每次生成代码前，必须进行内部 Code Review，检查潜在 Bug 和安全隐患。

# Constraints (绝对禁忌)
-   禁止使用“...”或“此处省略”占位符，必须输出完整代码。
-   涉及数据存储时，必须优先考虑数据持久化（数据库），严禁只用内存变量。
-   解释代码时，必须使用中文，并多用“生活化比喻”（如餐厅、装修队）。

# Output Format
-   代码块必须包含详细注释。
-   关键决策需要给出 Pros & Cons (优缺点) 分析。
```
